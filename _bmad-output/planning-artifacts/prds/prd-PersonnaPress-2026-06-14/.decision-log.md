## D-005: Database — Supabase Postgres
- **Decision:** Supabase Postgres replaces SQLite for all application data
- **Alternatives considered:** SQLite on Droplet (original architecture spec)
- **Rationale:** User directive. Managed Postgres removes the single-Droplet data ceiling, enables horizontal scaling, provides built-in connection pooling, and eliminates backup/replication concerns. The $6 Droplet still hosts FastAPI but no longer hosts the database.
- **Impact:** Removes the 50-user concurrency ceiling tied to SQLite single-writer. Changes infrastructure cost model (Supabase free tier → Pro at $25/mo). Eliminates the "disk failure = data loss" risk for media files.
- **Date:** 2026-06-14

## D-006: File Storage — Supabase Storage
- **Decision:** Supabase Storage for all file uploads (brand content) and generated assets (FLUX images)
- **Alternatives considered:** Nginx static file serving on Droplet (original architecture spec)
- **Rationale:** User directive. CDN-backed storage with public URLs, no Droplet disk dependency, built-in access control policies.
- **Impact:** Nginx no longer serves static media — its sole role is reverse proxy to Uvicorn. `media_url` in campaigns now points to Supabase Storage URLs instead of local Nginx paths.
- **Date:** 2026-06-14

## D-007: Persistent Job Records
- **Decision:** All scheduled jobs and generation tasks are persisted in the database. APScheduler reads from DB on startup so jobs survive process restarts.
- **Alternatives considered:** In-process ephemeral APScheduler (original architecture spec)
- **Rationale:** User directive. Addresses the critical gap where a process crash would lose scheduled publish jobs. Retries also use persistent records to track attempt count and failure state.
- **Impact:** Requires a `jobs` or `scheduled_tasks` table in Supabase Postgres. APScheduler configured with a persistent job store (SQLAlchemy-backed against Supabase Postgres).
- **Date:** 2026-06-14

## D-010: Fully Cloud Architecture (not local/hybrid)
- **Decision:** v1 is fully cloud-based. No local/on-device inference.
- **Alternatives considered:** Local/cloud-hybrid LLM agent (described in original concept brief `full.prd.md`)
- **Rationale:** Target users (founders, coaches, agency owners) are not ML engineers. Local inference requires hardware setup and adds operational complexity that conflicts with the "dump your idea, get a draft" value proposition. Privacy-sensitive users are not the v1 audience.
- **Impact:** Added to Non-Goals (§5) explicitly. Closes the silent gap from the concept brief.
- **Date:** 2026-06-14

## D-011: Voice Fidelity Check — Advisory Second-Call Scoring
- **Decision:** After blog generation, a second Gemini call scores the draft on tone (0–10), cadence (0–10), and banned-jargon violations (count). Passes at 7/6/0. Advisory — surfaces a warning badge but does not block approval.
- **Alternatives considered:** (a) Embedding similarity scoring (needs training data infrastructure), (b) Human A/B evaluation (doesn't scale), (c) No check (relies on prompt quality alone)
- **Rationale:** Provides signal without blocking workflow. Thresholds can be tuned per-Client in v2. If SM-3 (approval rate) falls below 70%, re-evaluate making the check blocking.
- **Impact:** Replaced the vague A-12 assumption. Now a defined mechanism in FR-13 with testable thresholds.
- **Date:** 2026-06-14

## D-012: GDPR — Policy-Only for v1 Launch
- **Decision:** GDPR compliance in v1 is scoped to Privacy Policy and Terms of Service disclosures. Technical compliance (data export API, right-to-deletion endpoint, cookie consent) deferred to v1.1.
- **Rationale:** Engineering cost of full GDPR technical compliance should not block initial launch for a North America-targeted product. Legal review of Policy/ToS is a launch blocker.
- **Date:** 2026-06-14

## D-013: WordPress Draft-First Publishing
- **Decision:** WordPress publishing uses a draft-first pattern: POST as draft, upload media, then PATCH to publish. Prevents broken posts from going live on partial failure.
- **Rationale:** Research recommendation. Resilience safeguard for the most common publishing target.
- **Date:** 2026-06-14

## Story

As a user publishing to GitHub Pages,
I want the committed post file to include the SEO meta description and relevant tags in the front matter,
so that my static site has proper SEO metadata without manual editing after publish.

## Acceptance Criteria

1. When publishing to any GitHub Pages framework (Jekyll, Astro, Next.js, Hugo, Eleventy), the front matter `description` field is populated from the blog post's `<!-- meta: ... -->` HTML comment rather than the empty `voice_score` lookup.
2. When publishing to any GitHub Pages framework, `tags` (or `categories` for Jekyll) are populated from `voice_score.tags` — a new field added to the fidelity check output.
3. Jekyll front matter always includes `categories` from `voice_score.tags` when available (already done, but verified against AC 1 fix).
4. Astro front matter always includes `tags` from `voice_score.tags` when available — the existing "only if template has tags key" gate is removed.
5. Next.js front matter always includes `tags` from `voice_score.tags` when available — the existing "only if template has tags key" gate is removed.
6. Eleventy front matter always includes `tags` from `voice_score.tags` when available — the existing "only if template has tags key" gate is removed.
7. Hugo front matter always includes `tags` from `voice_score.tags` when available (already done, verified).
8. The fidelity check Gemini prompt (`check_fidelity` in `gemini.py`) is updated to also return a `tags` field: a list of 3–5 concise SEO-relevant tags derived from the blog content.
9. The `check_fidelity` validation in `gemini.py` accepts (but does not require) a `tags` field — it does not raise if `tags` is missing, to maintain backward compatibility with existing campaigns that lack the field.
10. The `VoiceScore` TypeScript interface in `frontend/lib/types.ts` is updated to include `tags?: string[]`.
11. The front matter preview in `approval-panel.tsx` (`buildFrontMatterPreview`) shows the actual meta description (extracted from `campaign.blog_html`) and actual tags (from `campaign.voice_score?.tags`) instead of hardcoded empty strings.

## Dev Notes

### Root Cause: Why description was always empty

`publishing.py` uses `campaign.voice_score.get("meta_description", "")` to get the description. But `voice_score` is the **fidelity check result** from `gemini.check_fidelity()` — it only ever contains `tone_score`, `cadence_score`, `jargon_violations`, `seo_bluf_present`, `seo_h2_count`, `seo_faq_present`, `seo_fluff_detected`. There is no `meta_description` key. So description was always `""`.

The meta description IS generated — it's embedded in the blog HTML as an HTML comment on line 2, per the blog generation prompt:
```html
<h1>Title here</h1>
<!-- meta: One sentence meta description, max 150 chars, ends with action phrase -->
<div class="tldr">...
```

The `_extract_meta_description` regex approach is the correct fix: it extracts the description directly from the source of truth (the generated blog HTML).

### Root Cause: Why tags were always missing

Tags were never generated anywhere in the pipeline. The fidelity check prompt returns scoring data, not content metadata. The fix: add `tags` to the fidelity prompt output since that's the step where Gemini already reads the full blog HTML — extracting tags from the same model call costs zero additional API calls.

### Updated `buildFrontMatterPreview` spec

```tsx
function buildFrontMatterPreview(framework: string, title: string, description: string, tags: string[]): string {
  const now = new Date().toISOString().replace(/\.\d{3}Z$/, "Z");
  const safe = title.replace(/"/g, '\\"');
  const safDesc = description.replace(/"/g, '\\"') || "";
  const tagsYaml = tags.length > 0
    ? `[${tags.map((t) => `"${t.replace(/"/g, '\\"')}"`).join(", ")}]`
    : "";

  if (framework === "jekyll") {
    const categoriesLine = tagsYaml ? `categories: ${tagsYaml}\n` : "";
    return `---\nlayout: post\ntitle: "${safe}"\ndate: ${now}\ndescription: "${safDesc}"\n${categoriesLine}---`;
  }
  if (framework === "astro") {
    const tagsLine = tagsYaml ? `\ntags: ${tagsYaml}` : "";
    return `---\ntitle: "${safe}"\ndescription: "${safDesc}"\npubDate: "${now}"\nheroImage: ""${tagsLine}\n---`;
  }
  if (framework === "hugo") {
    const tagsLine = tagsYaml ? `\ntags: ${tagsYaml}` : "";
    return `---\ntitle: "${safe}"\ndate: ${now}\ndescription: "${safDesc}"\ndraft: false${tagsLine}\n---`;
  }
  // eleventy / nextjs / generic
  const tagsLine = tagsYaml ? `\ntags: ${tagsYaml}` : "";
  return `---\ntitle: "${safe}"\ndate: ${now}\ndescription: "${safDesc}"${tagsLine}\n---`;
}
```

### Astro tags placement — front matter line order

For Astro, insert `tags` after `heroImage` (not before `---`). The Astro schema expects: `title`, `description`, `pubDate`, `heroImage`, then optional extras. Inserting before the closing `"---"` in `fm_lines` correctly places tags after heroImage.

Exact insertion point in Astro block:
```python
# After building fm_lines = ["---", 'title: "..."', 'description: "..."', 'pubDate: "..."', 'heroImage: "..."']
# and adding any extra_required_fields:
if campaign.voice_score and campaign.voice_score.get("tags"):
    tags_list = campaign.voice_score["tags"]
    if isinstance(tags_list, list) and tags_list:
        tags_yaml = ", ".join(f'"{t.replace(chr(92), chr(92)+chr(92)).replace(chr(34), chr(92)+chr(34))}"' for t in tags_list)
        # Insert before the closing "---"
        fm_lines.insert(len(fm_lines) - 1, f"tags: [{tags_yaml}]")
```

Wait — the Astro `fm_lines` does NOT have `"---"` as its last element (unlike other frameworks). Looking at the existing code:
```python
fm_lines = ["---", 'title: "..."', ...]
fm_lines.append("---")  # <— this closes it
```
So `fm_lines[-1]` IS `"---"`. Use `fm_lines.insert(len(fm_lines) - 1, f"tags: [{tags_yaml}]")` to place tags before the closing `---`.

For Next.js and Eleventy, the same pattern applies — check the existing `fm_lines` structure in those blocks.

### YAML escape safety

The existing YAML escaping pattern used throughout publishing.py is:
```python
t.replace(chr(92), chr(92)+chr(92)).replace(chr(34), chr(92)+chr(34))
```
This replaces `\` with `\\` and `"` with `\"`. Use this same pattern for tag strings.

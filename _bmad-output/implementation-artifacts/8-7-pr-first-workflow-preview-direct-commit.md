# Story 8.7: PR-First Workflow, Preview UI & Direct Commit Option
<!-- epics.md reference: Epic 8, Story 8.6 (GitHub Blog Publishing Phase 2) -->

Status: done

## Story

As an authenticated user,
I want to review exactly where my post will be written and choose between a Pull Request or a direct commit before publishing to GitHub,
so that I never accidentally corrupt a live site by committing the wrong file to the wrong location.

## Acceptance Criteria

1. "Publish to GitHub" Secondary button appears in the Approval Gate sticky action footer for approved campaigns with a connected GitHub repo (`detected_framework` set). Both GitHub and other-platform buttons may be present simultaneously.
2. Clicking "Publish to GitHub" expands an inline pre-publish confirmation panel below the sticky footer (Paper background, 1px Border top, 24px padding, no shadow at rest) showing: repo name, target file path (JetBrains Mono 13px Ink), a "Show front matter" toggle revealing a code block (collapsed by default), and a two-card publish mode selector — "Open Pull Request" (Active card by default) vs "Commit directly" — plus a "Confirm and publish" Primary button and a "Cancel" text link.
3. In "Open Pull Request" mode, clicking "Confirm and publish" calls `POST /api/v1/campaigns/{id}/publish/github` with `{"mode": "pr"}`; FastAPI creates a branch `personnapress/{campaign_id_short}`, commits the post file, opens a PR titled with the H1, creates a `jobs` record with `job_type='github_publish'` and `status='pending'`; the confirm button shows an inline spinner while the job runs.
4. On successful PR creation: the pre-publish panel collapses; a confirmation line appears below the footer — "PR opened — [{title truncated to 45 chars}]({pr_url})" as an Ink-colored link; the Approval Gate header shows a new "PR OPEN" StatusBadge variant (Default Card fill: `#E5E5E5` border, Graphite text, uppercase tracked label "PR OPEN", 2px border-radius).
5. In "Commit directly" mode, calling `POST /api/v1/campaigns/{id}/publish/github` with `{"mode": "commit"}` commits directly to the default branch; `campaigns.status` transitions to `published`; the footer shows "Published to {repo_name}" with the 7-character commit SHA as a text link.
6. If the connection has `direct_commit_default: true` stored in its credential JSON, the pre-publish panel opens with "Commit directly" pre-selected as Active. A toggle on the GitHub connection card (Secondary button style, labelled "Default to direct commit") sets/clears this preference.
7. When a GitHub webhook fires (`pull_request.closed` with `merged: true`) for a PR URL matching `campaigns.github_pr_url`: the Next.js webhook route at `/api/webhooks/github` validates the payload signature using `GITHUB_APP_WEBHOOK_SECRET`; calls FastAPI `POST /api/v1/webhooks/github`; FastAPI matches PR URL to Campaign and transitions `campaigns.status` from `approved` to `published`; the Approval Gate shows "Published" on next load.
8. All interactive elements in the pre-publish panel are keyboard-navigable: mode selector cards respond to Tab + Space/Enter; "Show front matter" link has `aria-expanded`; file path block has `role="region" aria-label="Publish target file path"`; confirm button has `focus-visible:ring-2 focus-visible:ring-ink focus-visible:ring-offset-2`.

## Dev Notes

### Job Creation Before BackgroundTask (Critical)

Per the architecture pattern, the `jobs` record MUST be created before dispatching `add_task()`. The worker (`publish_github_job`) reads `job_id` to update status. If you dispatch without creating the record first, the worker may try to update a non-existent job.

```python
# CORRECT — job record first, then dispatch
job = await create_job(db, job_type="github_publish", status="pending", campaign_id=campaign_id)
background_tasks.add_task(publish_github_job, job.id, campaign_id, mode)
return {"job_id": str(job.id)}, 202
```

### Mode Selector Keyboard Accessibility (AC-8)

The two mode cards must behave as a radio group. Use `role="radiogroup"` on the container and `role="radio" aria-checked={isSelected}` on each card. Tab moves between them; Space/Enter toggles.

### Panel Expansion Below Sticky Footer

The sticky footer in the Approval Gate uses `position: sticky; bottom: 0`. The confirmation panel must be rendered INSIDE the sticky footer container (not outside it) so it scrolls up with the footer. If the panel is outside the sticky container, it will appear behind the footer on scroll.

### Front Matter Preview

The "Show front matter" code block should render the YAML/TOML that WILL be written — generate it client-side from the campaign data already loaded in the Approval Gate (title from H1, date from today, etc.). No additional API call needed for the preview. Use `<pre className="font-mono text-[12px] bg-[#F9F9F6] border border-[#E5E5E5] p-3 overflow-x-auto">`.

### GitHub API Version Header

All new calls in `integrations/github.py` (`create_branch`, `get_branch_sha`, `create_pull_request`) must use the module-level `GITHUB_HEADERS` constant with `X-GitHub-Api-Version: 2026-03-10`. This is the current stable API version (July 2026).

### Webhook Signature Verification

```python
# webhooks.py — GitHub signature pattern (current API version 2026-03-10)
import hmac
import hashlib

def verify_github_signature(payload: bytes, signature_header: str, secret: str) -> bool:
    expected = "sha256=" + hmac.new(
        secret.encode("utf-8"), msg=payload, digestmod=hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature_header)
```

Call this BEFORE processing the event. Return HTTP 400 if verification fails. `signature_header` is the value of the `X-Hub-Signature-256` request header.

### PR URL Storage

`campaigns.github_pr_url` is the `html_url` from the GitHub PR response (e.g., `https://github.com/user/repo/pull/42`). The webhook matches on exact URL equality. Store and compare the full URL exactly as returned by GitHub.
